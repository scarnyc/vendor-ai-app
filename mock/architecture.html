v0.6 graph deltas. (1) await_run is a new initial state — selecting a case in the canvas tabs loads metadata only; the graph blocks until the operator presses Run agent. This makes case selection cheap (no auto-LLM) and matches how procurement reviewers actually work. (2) The dashed edge from HITL back to classify_data_sensitivity is the edit-and-re-run loop — Priya's edits flow as forwardedProps, only the LLM-driven downstream nodes re-run, and the deterministic tools (budget · vendor · TCV · validate) short-circuit via memoized state because their inputs haven't changed.
SPEC §9 enforcement. The graph contains no edge that emits an external message, accepts contract terms, or marks spend approved without a Command(resume=…) from the human gate. draft_vendor_followup only writes a draft into the packet; sending requires Priya copying it out by hand (no Send button anywhere — see DESIGN.md §10). validate_citations rejects flags whose quoted policy text isn't a verbatim substring of the cited doc — closes the no-hallucination loop before HITL. The edit-and-re-run loop preserves these constraints because it re-enters the same gate: any state mutation that would emit, accept, or finalize must still pass through HITL on the next pass.

State + checkpointer

LangGraph.js graph compiled with MemorySaver (Vercel ephemeral filesystem). Thread keyed by ?case=001&thread=<uuid> URL params so a refresh after timeout resumes cleanly. Productionization swaps to a Postgres checkpointer for durable HITL state across cold starts.

All nodes stream — Vercel Hobby's 10s function timeout resets on every chunk, so a long completion stays alive as long as bytes flow.

Memoization for the re-run loop: deterministic tool results (lookup_budget, check_existing_vendor, calculate_total_contract_value, validate_required_documents) are stored in checkpointer state keyed on their inputs. The edit-and-re-run edge re-enters at classify_data_sensitivity; preceding tool nodes are skipped when their input fingerprint matches.

Operator vs. recipient (SPEC §9 routing)

Only Procurement can resolve await_run or the HITL gate. The 6 recipient lenses (Business Owner · Legal · Security · VP Finance · CFO · Executive) render the same packet filtered to their domain, with action buttons hidden — there is no graph edge a recipient can trigger. Routing happens after Priya approves; recipients act in their own systems (Slack, email, Workday), never inside Vendor AI.

This is a deliberate UX-and-graph choice, not a permissions hack. See DESIGN.md §2 for the full permission table and rationale.
